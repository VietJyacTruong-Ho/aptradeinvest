## Design Context

### Users

**Primary audience is dual-use:** internal APBAI staff, researchers, and leadership using the dashboard for ongoing analysis and decision-making; and external stakeholders (investors, government officials, journalists, academics) who encounter it as a public-facing resource representing the organization. The internal user runs long analytical sessions; the external user skims for credibility and key figures. Both audiences expect the interface to signal institutional authority.

### Brand Personality

**Authoritative · Precise · Institutional**

This is a think-tank/government data portal aesthetic. Credibility is the primary design value — every visual decision should reinforce that the data and the organization behind it can be trusted. Restraint earns trust here more than expressiveness does.

**Voice & tone:** Measured, factual, confident. No jargon-for-jargon's sake, but no oversimplification either. The interface speaks fluent "policy and commerce."

### Tech Stack & Existing Tokens

- **Fonts:** `Libre Baskerville` (display/headings, serif authority), `IBM Plex Sans` (body/UI, clean and legible), `IBM Plex Mono` (data values, numbers, timestamps)
- **Color palette:**
  - Deep navy: `#0d1b2a` (header, primary text)
  - Background: `#f2f4f7` (light neutral)
  - Card surface: `#ffffff`
  - Border: `#dde2ea`
  - Muted text: `#536070`; Hint text: `#96a1ae`
  - Blue accent: `#378ADD`; Info/link: `#1655a2`
  - Financial value amber: `#BA7517`
  - Status palette: info (`#1655a2`), success (`#1a6b3c`), warning (`#8a5a0a`), danger (`#9b1f22`), planned (`#5b21b6`)
- **Radii:** `--r: 6px` (cards), `--rs: 4px` (small elements)
- **Framework:** Vanilla HTML/CSS/JS — no component library or build tool

### Aesthetic Direction

**Visual tone:** Dense but orderly. The dashboard packs meaningful data into a compact viewport without feeling cluttered. Whitespace is used deliberately, not generously. Charts and numbers are the visual protagonists; chrome (nav, labels, controls) should be quiet.

**References (implied):** OECD.stat, Federal Reserve FRED, IBISWorld, well-designed government data portals.

**Anti-references:** Startup dashboards with gradient cards and big illustrative icons. Consumer analytics tools (Mixpanel, Amplitude aesthetic). Anything that looks like a marketing dashboard or a PowerPoint.

**Theme:** Light mode only (for now). Print/export and mobile responsiveness are on the roadmap — design choices should degrade gracefully to static, paginated output.

### Roadmap Considerations

- **Export/print views** are planned — avoid layout choices that rely purely on viewport geometry or JS-rendered animations. Design components to have a print-safe fallback.
- **Mobile/responsive** is planned — use fluid-first column patterns where possible. The current overflow:hidden full-viewport layout will need rethinking for smaller screens.
- **Map visualization** is under consideration — leave visual vocabulary room for a geographic component (potentially a choropleth of Indiana counties or Asia-Pacific origin countries).
- **More data cards/visualizations** may be added — maintain consistent density norms and a clear visual hierarchy so new components don't disrupt existing rhythm.

### Design Principles

1. **Data is the hero.** Every chrome element (headers, filters, labels, borders) should recede to serve the numbers and charts. When in doubt, remove decoration rather than add it.

2. **Earn trust through precision.** Typography, spacing, and color should feel deliberate and calibrated — not approximate. Mono fonts for numbers, exact alignment, restrained color use. Sloppiness erodes credibility in an institutional context.

3. **Serve two modes simultaneously.** The internal analyst needs density and depth; the external stakeholder needs a fast read. Design for scannable hierarchy (big numbers, clear labels) that also rewards close reading (tooltips, sourcing, fine-grained filters).

4. **Design for the page, not just the screen.** Since print/export views are planned, avoid effects that only work in browser (box-shadows as primary information carriers, viewport-locked layouts, motion-only states). Every view should have a static equivalent that holds up on paper.

5. **Expand without crowding.** The project will grow — more countries, a map, more chart types. Establish and defend spacing norms and a consistent visual grammar so new features slot in without disrupting the existing rhythm.
